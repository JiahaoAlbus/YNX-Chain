# Migration and compatibility

The market store uses an integrity envelope and market schema version 5. Versions 1–4 migrate in memory to version 5: version 1 gains an empty worker-key registry, version 3 adds optional provider continuity evidence and migration-target fields, version 4 adds empty auction/sealed-bid registries and optional quote auction references, and version 5 adds an empty erasure-request registry. No worker identity, signed meter, maintenance event, exit, auction, bid, quote or privacy request is invented. The next successful mutation persists version 5. Digest mismatch and unsupported schema versions fail closed. Atomic write, fsync, rename and last-snapshot backup are implemented by `internal/productstore`.

The recovered pool store and the new provider market store are intentionally separate. No automatic conversion treats legacy pools, credits, quotes or income as verified provider capacity, metering or settlement. A future importer must produce a preview, counts, source digest, per-record decision and rollback file before operator approval.

Old clients may continue using recovered `/api/state` and `/api/actions`. New provider-market clients use `/api/market/*`. Deprecation requires telemetry-backed notice, at least one candidate release overlap and an export path. Legacy Wallet challenge routes remain `410`; canonical Wallet authorization is the only supported authentication migration.

Exact-schema market backup recovery is covered by `TestMarketBackupRestoreDrill` and `go run ./apps/resource-market/cmd/recover -kind market -store <path>`. The drill proves corruption rejection and restoration of the previous atomic snapshot. Authenticated users can download actor-scoped records at `GET /api/market/export`. Erasure is independently fulfilled only after live obligations clear; it pseudonymizes identity while preserving economic, dispute, receipt and audit integrity. Retention policy `resource-market-retention-v1` removes expired unaccepted quotes after 30 days and rejected/failed auction transients after 90 days. A final source-commit-bound drill artifact, rollback migration and deployed scheduler/legal approval remain required before `testedLocal` can become true.
