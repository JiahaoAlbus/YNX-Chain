# API Reference

Core: `GET /health`, `GET /status`, `GET /node/identity`, `GET /metrics`, `GET /blocks/latest`, `GET /txs/{hash}`, `GET /accounts/{address}`, `GET /validators`, `GET /validators/peers`, `GET /validators/peer-sync`, `POST /validators/{address}/heartbeat`, `POST /validators/{address}/peers/observe`, `POST /validators/{address}/peer-sync`.

`GET /status` includes `validatorCount`, `readyValidatorCount`, `validatorPeerReadiness`, `validatorPeerDiscovery`, `validatorPeerSync`, `nodeIdentity`, and `build` so node operators and public proof checks can distinguish configured validators, locally ready validators, expected bootstrap peers, observed peers, recorded sync evidence, the configured local validator identity, peer-sync freshness, and the non-secret binary release identity serving the endpoint.

`GET /node/identity` returns non-secret runtime identity evidence for the local node: `validatorAddress`, validator moniker/role/host/peer ID when configured, `expectedValidatorCount`, `peerSyncTargetCount`, target validator addresses, `peerSyncInterval`, `build`, and `peerSyncFreshness`. `build` contains `commit`, `release`, and `buildTime`; local development defaults are `unknown`, `local`, and `unknown`, while deploy builds inject `ynx-chaind` release identity through Go linker flags. Freshness records classify each configured target as fresh, stale, missing, synced, or lagging using persisted `/validators/peer-sync` records. This endpoint is designed for post-deploy proof and does not expose private keys, secrets, or full env files.

Deploy bundles also include `config/release-manifest.json` with SHA-256 checksums for binaries, role env files, and service config. Public proof expects `/status.build` and `/node/identity.build` to match that manifest's `commit` and `release`, and remote SSH verification compares the installed `ynx-chaind` checksum to the manifest before treating the node as release-identifiable.

`GET /validators` returns `validators` plus `expectedValidatorCount`. Validator records include `address`, `moniker`, optional `host`, optional `role`, optional `peerId`, `votingPower`, `active`, `peerReady`, optional `peerStatus`, optional `latestHeight`, optional `lastSeenAt`, optional `updatedAt`, and optional `peerEvidence`. Production/testnet nodes load configured validator identity from `YNX_VALIDATOR_SET`; if it is missing, the node exposes only the local default validator and remote public proof must fail multi-validator checks. Runtime peer readiness is persisted in the local chain snapshot and survives restart for matching validator addresses.

`POST /validators/{address}/heartbeat` records local validator peer-readiness evidence for an existing validator. Request body accepts optional `peerId`, optional `host`, optional `ready`, optional `status`, optional `latestHeight`, and optional `evidence`. If `ready` is omitted, the heartbeat marks the validator ready. This is operational readiness evidence for the local node and does not claim remote public proof until the same validator set is deployed and verified through public endpoints.

`GET /validators/peers` returns persisted validator peer discovery records with `address`, optional `peerId`, optional `host`, optional `p2pAddress`, optional `role`, `expected`, `observed`, `status`, `source`, optional `latestHeight`, optional `lastSeenAt`, optional `updatedAt`, and optional `evidence`. Nodes derive expected peers from `YNX_VALIDATOR_SET` and may receive explicit bootstrap peer records from `YNX_BOOTSTRAP_PEERS` using `address|peerId|host|p2pAddress|role` entries or a JSON array.

`POST /validators/{address}/peers/observe` records observed peer discovery evidence for an existing validator. Request body accepts optional `peerId`, optional `host`, optional `p2pAddress`, optional `status`, optional `latestHeight`, and optional `evidence`. This creates restart-safe local peer-discovery evidence and is required by public-proof checks after remote deployment, but it is not itself proof that remote public validators are live.

`GET /validators/peer-sync` returns persisted peer sync records with `id`, `source`, `target`, `sourceHeight`, `targetHeight`, `lagBlocks`, `status`, optional `evidence`, `createdAt`, and `updatedAt`.

`POST /validators/{address}/peer-sync` records local sync evidence from the source validator in the path to another configured validator. Request body accepts `target`, optional `sourceHeight`, optional `targetHeight`, optional `status`, and optional `evidence`. If `status` is omitted, the node classifies lag within one block as `synced` and larger lag as `lagging`. Source and target must both be configured validators and cannot be the same address. This is restart-safe local sync evidence for deployment readiness and public-proof checks, not a claim that remote node-to-node transport has already been verified.

`ynx-chaind` can also derive peer height-observation records automatically when `YNX_LOCAL_VALIDATOR_ADDRESS`, `YNX_PEER_RPC_URLS`, and optional `YNX_PEER_SYNC_INTERVAL` are set. `YNX_PEER_RPC_URLS` accepts `targetValidator|http://host:port` entries separated by semicolons or a JSON array of `{ "address": "...", "url": "..." }` objects. The node polls each target `/status`, records local source height, target height, lag, and `peer-rpc-poll:<url>/status` evidence, then persists the result through `/validators/peer-sync`. Fresh records, including `fresh_with_lag`, prove current peer reachability and height observation; they do not prove consensus, block replication, or state convergence.

Authoritative testnet replication uses one configured producer and read-only followers. The producer enables `YNX_BLOCK_PRODUCTION_ENABLED`, serves `GET /internal/replication/snapshot` only to callers authenticated by `X-YNX-Replication-Key`, and signs the exact response bytes with HMAC-SHA256 in `X-YNX-Replication-SHA256`. Followers disable block production, configure `YNX_REPLICATION_SOURCE_URL`, verify the response HMAC, validate network identity, contiguous heights, parent hashes, block identities, and transaction/block linkage, then atomically replace and persist replicated chain state. Follower mutation requests return HTTP `409`. `GET /node/identity` exposes `blockProductionEnabled`, `replicationMode`, and `replicationSource` without exposing the key. This provides deterministic single-chain state convergence and restart recovery; it is not BFT consensus, validator voting, or Byzantine fault tolerance.

`GET /ynx/overview` on `ynx-indexerd` returns the indexed network, chain ID, native currency symbol, latest and earliest-retained source heights, earliest source hash/time, indexed height, block/transaction counts, last sync state, build identity, and truthful service status. It is the public overview contract used by remote verification.

`ynx-chaind --check-config` validates the same node runtime env used by service startup without binding HTTP ports, writing chain state, producing blocks, or starting peer polling. It parses `YNX_VALIDATOR_SET`, `YNX_BOOTSTRAP_PEERS`, `YNX_LOCAL_VALIDATOR_ADDRESS`, `YNX_PEER_RPC_URLS`, and `YNX_PEER_SYNC_INTERVAL`, then fails unsafe multi-validator testnet configs such as missing local identity, local identity outside the validator set, incomplete peer target coverage, self targets, duplicate targets, or targets outside `YNX_VALIDATOR_SET`. Deployment dry-runs validate every generated role-specific env with this command, and real remote deployment runs it after installing `/etc/ynx/ynx-chaind.env` but before restarting `ynx-chaind`.

`ynx-chaind --export-consensus-state <path>` loads the configured persistent state without starting HTTP, block production, replication, or peer polling, then atomically writes a mode-`0600` BFT migration file. The export binds network identity, last committed height/hash, sorted YNXT account balance/stake/nonce/resource/lot state, sorted core validator voting state, resource policy, liquid/staked totals, and a SHA-256 state hash. Operational peer heartbeat/readiness timestamps are deliberately excluded from the consensus hash. `make consensus-migration-check` creates persisted activity, stops the node, exports the file, and independently verifies the commit point, YNXT receiver balance, validator set, and state hash. This is a deterministic migration boundary, not evidence that BFT consensus is already implemented.

`ynx-abci -migration-state <path> [-state <path>]` starts the YNX CometBFT ABCI 2.0 application over `socket` (default `tcp://127.0.0.1:26658`) or `grpc`. The default durable state path is `<migration-state>.abci-state.json`. It validates the migration anchor before listening, binds `InitChain` to `ynx_<numeric-chain-id>-1`, requires the initial height to continue after the migrated commit, and exposes `/migration`, `/state`, `/accounts/{address}`, AI, Pay, Resource policy/quote/delegation/rental/income/analytics, governance, Trust, and transparency queries. Canonical native transfers and domain-separated `application_action` envelopes use EVM-compatible secp256k1 ownership, chain ID, fixed YNXT fees, and one shared account nonce. AI, Pay, Trust, and Resource actions bind typed canonical payloads; Resource actions charge only shared fee/bandwidth and settle policy-bound YNXT without AI/Pay/Trust credits. Trust governance creation reuses the canonical Request Validity classifier; illegal, overbroad, evidence-free, unsupported-asset, and direct native YNXT freeze requests persist as rejected evidence instead of executing asset action. Resource delegation/rental state preserves liquid-plus-staked supply, quote expiry/policy identity, provider/protocol settlement, event/idempotency linkage, and exact stake reconciliation. `CheckTx` verifies ownership and state constraints; proposal handlers execute transactions sequentially and reject invalid or duplicate nonce order; `Commit` atomically stores mode-`0600` state before advancing memory. `make bft-ai-action-check`, `make bft-pay-action-check`, `make bft-trust-action-check`, and `make bft-resource-action-check` verify signed mutation paths, persistence, concurrent nonce safety, and Gateway evidence mismatch rejection. Complete Trust and Resource surfaces passed fresh remote four-application/four-signer candidate proof and rollback and are reported as implemented; neither is persistently deployed or publicly routed, and this is not public BFT proof.

The BFT Gateway supports `eth_getTransactionByHash` and `eth_getTransactionReceipt` for committed native transfers and signed application actions. It verifies canonical transaction hash, Comet `/tx` proof response (including Comet's structured HTTP-500 not-found envelope), transaction payload hash, block membership, block hash/height, transaction index, `gas_used`, and cumulative gas from `/block_results`; unknown transactions return `null`, application actions return a null recipient, and inconsistent evidence fails closed. `eth_getLogs` validates a retained, at-most-1000-block committed range plus canonical address/topic filters. Current BFT envelopes do not execute EVM `LOG` opcodes, so the only truthful BFT log result is an empty array; local devnet contract logs are not copied or fabricated. `make bft-evm-receipt-check` verifies this boundary. Native/action receipts, empty logs, malformed/outside-history/overbroad rejection, four-application equality, four-signer recovery, cleanup, and rollback passed fresh private candidate proof; the capability is implemented but not persistently/publicly routed, and this is not public BFT proof.

`ynx-consensus-lab -ephemeral -local-fixture -output <new-dir>` creates a disposable four-validator local network. For a real migration input, replace `-local-fixture` with `-migration-state <path>`. The generator binds each YNX validator identity to its CometBFT ed25519 public key and consensus address, recalculates the migration hash, emits a common validated genesis and non-secret network manifest, and writes private validator/node keys only as mode-`0600` files. `ynx-consensus-tx -key <raw-32-byte-mode-0600-key> -to <address> -amount <YNXT> -nonce <next>` emits a canonical signed transaction without printing the key. `make consensus-quorum-check` exercises these commands against four real CometBFT and ABCI processes, broadcasts a signed transfer, removes one validator while quorum continues, then restarts and catches it up. These commands prove local behavior only; generated lab keys are forbidden for remote or custody use.

`ynx-consensus-package` validates an exported migration plus a schema-bound, public-key-only four-validator manifest and generates a hash-covered production candidate package. It requires an explicit UTC genesis time and a new output directory. `-verify-package <dir>` rechecks the complete hashed file set and rejects unsafe or private-key filenames. `ynx-consensus-keycheck` runs on each validator host and verifies that mode-restricted CometBFT private validator and node key files derive exactly the public key, consensus address, and node ID approved for that role. The candidate package fixes RPC to `127.0.0.1:27757`, ABCI to `127.0.0.1:27858`, metrics to `127.0.0.1:27660`, P2P to the approved RFC1918 endpoint, and state to `/var/lib/ynx-chain/consensus-candidate`. It does not modify public RPC routes or the authoritative `ynx-chaind` service.

EVM JSON-RPC: `POST /evm` supports `eth_chainId`, `net_version`, `eth_blockNumber`, `eth_getBalance`, `eth_getTransactionByHash`, `eth_getTransactionReceipt`, `eth_sendRawTransaction`, `eth_sendTransaction`, `eth_estimateGas`, `eth_call`, `eth_getLogs`, `eth_getBlockByNumber`, and `eth_getBlockByHash` in local devnet form. Receipts include persisted EVM-style logs for known transactions, including local contract deployment logs derived from stored contract event metadata and bounded ERC20 transfer logs derived from local contract-call execution. `eth_getLogs` filters persisted logs by `fromBlock`, `toBlock`, `address`, and `topics`. `eth_call` supports source-analyzer pure/view literals for ad hoc IDE snippets and a narrow Hardhat artifact staticcall subset where ABI selectors are generated by `ethers`, recorded in `artifacts/ynx-selector-metadata.json`, required to appear in the solc deployed bytecode, and then executed through a bounded read-only local EVM opcode interpreter for supported static getters such as ERC20 `decimals()`, constructor-seeded `totalSupply()`, and constructor-seeded `balanceOf(address)` mapping reads. Full calldata is accepted for selector matching, and the local interpreter supports Ethereum legacy Keccak `SHA3` for `mapping(address=>uint)` storage reads. `eth_sendTransaction` now supports a generic pinned-artifact write-call subset for nonpayable/payable calldata that fits the bounded local EVM state-transition interpreter. Current proof covers ERC20 `transfer(address,uint256)` and `SampleEVMWriteCounter.increment(uint256)`: the interpreter runs solc deployed bytecode, records `SSTORE` writes, captures `executionLogs` from bounded `LOGn` opcodes with Hardhat/ethers event topics, updates canonical local runtime storage slots, returns bounded ABI output when the path is supported, and creates a `contract_call` transaction. ERC20 transfer logs and supported generic bytecode logs such as `CountChanged(address,uint256)` are materialized into local receipts and filterable `eth_getLogs` results after block production. This is still a bounded local execution path, not full EVM state transition support, complex dynamic storage layout support, arbitrary opcode coverage, remote deployment, or remote public proof.

Indexer service: `ynx-indexerd` reads the chain RPC, persists indexed blocks and transactions, and exposes `GET /health`, `GET /metrics`, `POST /sync`, `GET /blocks/latest`, `GET /blocks/{height}`, `GET /txs`, and `GET /txs/{hash}` on the indexer HTTP port. For migration-height sources, an empty database starts at the source's exact `earliestBlockHeight`; resume requires parent-hash continuity with the stored tip and fails with rebuild guidance when the resume point was pruned, the source height regressed, or earliest/latest hashes disagree. Sources without an explicit earliest boundary preserve the authoritative height-`0` behavior.

Explorer service: `ynx-explorerd` reads both RPC and indexer sources and serves the reviewer-facing Explorer web/API surface. It exposes `GET /health`, `GET /metrics`, `GET /api/summary`, `GET /api/stream`, `GET /api/blocks/latest`, `GET /api/blocks/{height}`, `GET /api/txs`, `GET /api/txs/{hash}`, `GET /api/accounts/{address}`, `GET /api/tokens/YNXT`, `GET /api/validators`, `GET /api/resources/{address}`, `GET /api/resource-market/analytics`, `GET /api/fees/{hash}`, and `GET /api/search?q=...`. `GET /api/stream` is an SSE endpoint that emits a `dashboard` event every two seconds containing current summary, latest blocks, latest transactions, validator state, and resource-market analytics; it disables proxy buffering and reports core upstream failures as `upstream-error` events. Validator/resource enrichment failures are reported as snapshot warnings and do not interrupt block/transaction streaming. The Explorer authenticates its server-to-server resource request with `YNX_RESOURCE_GATEWAY_UPSTREAM_KEY`; that key is never returned to the browser. The web UI consumes this stream directly and falls back to ten-second snapshots after a stream failure. Wallet metadata reports native currency `YNXT`.

Faucet service: `ynx-faucetd` is the public faucet backend. It validates addresses, enforces per IP/address rate limits, writes a mode-`0600` JSONL request log, and exposes `GET /health`, `GET /metrics`, `POST /request`, and `POST /faucet`. Explicit `authoritative` mode preserves the privileged legacy RPC funding path. Explicit `bft` mode derives and checks the configured Faucet address from a process-local canonical hex or mode-restricted raw key, serializes account nonce selection, signs chain-`6423` YNXT transfers locally, submits the canonical envelope through `POST /transactions/broadcast`, and verifies committed hash/sender/recipient/amount/fee/nonce/height before reporting success.

`GET /ide/compiler` returns the pinned Solidity compiler configuration used by local IDE artifact and verifier metadata: Solidity `0.8.24`, wasm preference, optimizer settings, config hash, artifact kind, compiler mode, verifier mode, and current limitations.

`POST /ide/compile` returns deterministic local artifact metadata: source hash, bytecode hash, deployed bytecode hash, artifact hash, artifact kind, compiler config hash, compiler execution status, compiler mode, runtime mode, verifier mode, reproducibility status, deployed bytecode comparison status, ABI, parsed events, functions, selector source, `bytecodeSelectorMatched`, local runtime storage seed, local runtime storage slot metadata, and truthful limitations. `POST /ide/deploy` accepts optional `constructorArgs` for local verifier/devnet preflight and preserves them in contract metadata. `POST /ide/verify` and `GET /contracts/{address}` preserve that metadata for developer tooling. `POST /ide/call` and EVM `eth_call` return EVM-style encoded results and expose `executionStatus`, `executionEngine`, `opcodeStepCount`, `artifactKind`, and `bytecodeSelectorMatched` so callers can distinguish source-analyzer results from Hardhat artifact bytecode-interpreter staticcall subset results. `POST /ide/execute` accepts `caller`, `address`, and ABI `calldata` for a bounded local execution path; today it supports a generic pinned-artifact write-call subset for supported nonpayable/payable functions such as ERC20 `transfer(address,uint256)` and `SampleEVMWriteCounter.increment(uint256)`, returns `executionStatus=bounded_local_evm_sstore_state_transition_subset`, exposes `storageWrites` and `executionLogs`, updates canonical local runtime storage via bytecode-subset `SSTORE`, and creates a pending `contract_call` transaction. ERC20 Transfer logs and supported generic bytecode logs become filterable after block production.

For ad hoc IDE snippets, artifact kind remains `source-analyzer-artifact`. If the submitted source exactly matches a repository contract that has a Hardhat artifact built with the pinned Solidity `0.8.24` config, artifact kind upgrades to `pinned-solc-bytecode-artifact` and includes real bytecode/deployedBytecode hashes, ABI functions, `ethers` Keccak selectors, selector-presence checks against the solc deployed bytecode, runtime storage slot metadata, and a local storage seed for simple public primitive state plus constructor assignments such as `totalSupply = initialSupply`, `balanceOf[msg.sender] = initialSupply`, and `SampleEVMWriteCounter.count = initialCount`. `POST /ide/verify` then records `deployedBytecodeComparisonStatus=matched_local_deployed_bytecode_hash`. `POST /ide/execute` and `eth_sendTransaction` can use that metadata for the bounded local write-call path. This is still local devnet evidence, not remote public explorer proof or a complete EVM implementation.

`GET /ide/verifier/{address}` returns explicit local verifier service evidence for a deployed contract: compiler identity, source hash, bytecode hash, deployed bytecode hash, compiler config hash, artifact kind, compiler execution status, verifier mode, reproducibility status, deployed bytecode comparison status, local service status, remote public proof status, and limitations. It deliberately reports `remotePublicProofStatus=not_remote_public_proof` until a real remote verifier or explorer-backed proof is available.

Governance and Trust request safety:

- `ynx-trustd` is the independent public Trust and Chain Law API on port `6431`. Protected `/trust/*` and `/governance/*` routes require `X-YNX-Trust-Key` or `Authorization: Bearer <YNX_TRUST_API_KEY>` and return a unique `X-Request-ID`.
- The gateway enforces a 1 MiB request-body limit, a 2 MiB evidence/export response limit, per API-key/IP rate limits, and fail-closed pre-forward audit. Deployed `ynx-chaind` rejects direct Trust/Governance bypass without `YNX_TRUST_GATEWAY_UPSTREAM_KEY`.
- Gateway JSONL audit stores request metadata, body hashes, status, outcome, and audit hashes. It does not store request bodies, API keys, upstream keys, private keys, seed phrases, appeal evidence, or reviewer notes.
- `POST /governance/requests` stores a lawful-request intake record and classifies it as `VALID_UNDER_YNX_CHAIN_LAW`, `INSUFFICIENT_EVIDENCE`, `OUT_OF_SCOPE`, `OVERBROAD`, `ILLEGAL_OR_ABUSIVE`, `REQUIRES_GOVERNANCE_REVIEW`, `REQUIRES_USER_NOTICE`, or `REJECTED`.
- `GET /governance/requests/{id}` retrieves the persisted request, including classification, status, rule IDs, timestamps, and transparency entry ID.
- `POST /governance/requests/{id}/review` records explicit governance review status and `reviewedAt`; `POST /governance/requests/{id}/reject` records reviewer rejection reason, `REJECTED` classification, `rejectedAt`, and a transparency entry.
- `GET /governance/request-validity-rules` returns the named Chain Law / Request Validity rules used by governance and tracking review classification. Request and tracking-review responses include `ruleIds` so a reviewer can inspect why a classification was reached.
- Native `YNXT` requests cannot directly transfer, freeze, seize, blacklist, or bypass user signatures. Illegal, overbroad, evidence-free, off-scope asset-boundary, hidden-record, fake-risk, unsupported-Trust-conclusion, and AI-auto-punishment requests are rejected and recorded in the transparency report.
- Evidence-backed notification-only requests are classified as `REQUIRES_USER_NOTICE` with `notice_required` status; they create transparency records but do not perform asset action.
- `POST /trust/labels` records advisory-only address or transaction risk metadata including `labelId`, `subject`, `subjectType`, `address`, `labelType`, `severity`, `riskWeightBps`, `confidenceBps`, `source`, `evidenceHash`, `expiresAt`, `reviewRequired`, `appealAvailable`, `disputeStatus`, `legalStatusUnderYnxChainLaw`, optional rejected-request reference, and `assetEffect`. `subjectType` defaults to `address`; `transaction` requires an existing 32-byte transaction hash and leaves `address` empty. `assetEffect` must remain `none_advisory_only`; labels do not freeze, seize, confiscate, transfer, or classify users as criminals.
- `POST /trust/evidence` and `GET /trust/evidence/{id}` return `riskSummary` with effective advisory risk weight, active/expired/low-confidence/correction label counts, non-conclusive label IDs, active evidence hashes, reviewer notes, appeal path, and `assetEffect`. Expired labels and labels below 5000 confidence bps are kept for audit context but excluded from active risk scoring. Appeal correction labels created by false-positive decisions must remain visible through `riskSummary.correctionLabelCount` and reviewer notes.
- `POST /trust/appeals` opens an appeal record for false-positive correction or dispute review and also creates a transparency entry. Appeals must reference an existing governance/tracking-review `requestId` or an existing Trust `labelId`; missing or mismatched references are rejected so transparency entries cannot be created from orphaned appeals.
- `GET /trust/appeals/{id}` retrieves the persisted appeal, including request linkage, appellant, evidence, status, reviewer/decision fields when resolved, and transparency entry ID.
- `POST /trust/appeals/{id}/resolve` records reviewer, decision, updated status, and resolution reason. Decisions include `UNDER_REVIEW`, `NEEDS_MORE_EVIDENCE`, `ACCEPTED`, `REJECTED`, `LABEL_REMOVED`, and `LABEL_REDUCED`; accepted/removal/reduction decisions add corrective Trust labels rather than freezing or moving funds. Remote public proof must show that the correction is reflected in Trust evidence summary after resolution.
- `POST /trust/tracking-reviews` records purpose-limited tracking review metadata: requester, subject, purpose, query type, scope, evidence, institutional/sensitive flags, minimum-necessary status, confidence, expiry, classification, and appeal path. Overbroad, evidence-free, sensitive-inference, low-confidence punitive, or audit-bypass tracking requests are rejected or routed to review.
- `GET /governance/transparency` returns the locally persisted transparency report entries and counts. Remote public proof must use the public endpoint, not localhost.

Resource Market API safety:

- `ynx-resourced` is the independent public Resource Market API on port `6432`. Protected `/resource-market/*` routes require `X-YNX-Resource-Key` or `Authorization: Bearer <YNX_RESOURCE_API_KEY>` and return a unique `X-Request-ID`.
- The gateway enforces a 1 MiB request-body limit, a 2 MiB response limit, per API-key/IP rate limits, and fail-closed pre-forward audit. Deployed `ynx-chaind` rejects direct Resource Market bypass without `YNX_RESOURCE_GATEWAY_UPSTREAM_KEY`.
- Gateway JSONL audit stores request metadata, body hashes, status, outcome, and audit hashes, but not request bodies or credentials. Canonical policy, quote, delegation, rental, provider/protocol income, and analytics state remains in the persistent chain runtime.
- `GET /resource-market/policy`, `GET /resource-market/quote`, `POST /resource-market/delegations`, `GET /resource-market/delegations/{address}`, `POST /resource-market/rent`, `GET /resource-market/income/{address}`, and `GET /resource-market/analytics` are the authenticated public Resource Market surface.
- `YNX_RESOURCE_GATEWAY_UPSTREAM_MODE=authoritative` preserves the current rollback-compatible proxy. Candidate `bft` mode requires chain ID `6423`, a canonical signer address, and exactly one process-local raw private-key source, preferably a mode-`0600` file. Mutations require `idempotencyKey`; the daemon injects the signer as delegation provider or rental address, reads the committed policy and quote, serializes nonce allocation, signs a canonical action locally, verifies the committed delegation/rental response, and returns the existing public response shape. Exact replay returns the original committed object without another nonce or fee; changed-input reuse returns `409`.
- BFT delegation converts signer liquid YNXT to beneficiary staked YNXT. BFT rental transfers the policy quote from renter to provider and protocol treasury, persists the quote commitment, rental, income, event, and idempotency records in AppHash, and reduces only already-used resource counters. The fixed action fee is `1 YNXT`; Resource actions do not consume AI, Pay, or Trust units. Total liquid plus staked YNXT must remain equal to the migration anchor.

Pay merchant safety:

- `ynx-payd` is the independent public merchant API on port `6430`. Its protected `/pay/*` routes require `X-YNX-Pay-Key` or `Authorization: Bearer <YNX_PAY_API_KEY>` and return a unique `X-Request-ID`.
- `POST /pay/intents`, `POST /pay/invoices`, `POST /pay/refunds`, and `POST /pay/webhook-signatures` require `idempotencyKey` at the public gateway. Reusing the same key in the same merchant/object scope returns the original object instead of creating a conflicting duplicate. The authenticated merchant identity is injected from `YNX_PAY_MERCHANT_ID`; a conflicting client merchant is rejected.
- `POST /pay/webhook-signatures` stores replay-auditable metadata: `eventId`, `eventType`, `payloadHash`, `algorithm`, `signedAt`, `idempotencyKey`, and `replaySafe`. The HMAC key is loaded from `YNX_PAY_WEBHOOK_SIGNING_KEY`; clients cannot supply it, and it is never returned or stored in response or audit objects.
- `GET /pay/webhook-signatures/{eventId}` returns a stored signature for audit/replay verification.
- `GET /pay/events?intentId=...` and `GET /pay/events/{id}` return persisted payment audit events with `auditHash`, object id, amount, currency, merchant, and idempotency key metadata.
- `ynx-payd` enforces a 1 MiB request-body limit, per API-key/IP rate limits, fail-closed pre-forward access audit, and a dedicated upstream secret. Deployed `ynx-chaind` rejects direct `/pay/*` bypass attempts without `YNX_PAY_GATEWAY_UPSTREAM_KEY`. JSONL access audit contains only request metadata and body hashes, never bodies or credentials.
- `YNX_PAY_GATEWAY_UPSTREAM_MODE=authoritative` preserves the current rollback-compatible chain proxy. Candidate `bft` mode requires chain ID, a configured signer address, and exactly one process-local raw key source. It checks committed idempotency before nonce allocation, signs canonical Pay actions locally, sends only signed envelopes to the loopback BFT Gateway, verifies committed transaction/object evidence, and computes webhook HMAC locally with a separate key. Identical replay returns the committed object without another nonce or fee; changed input fails with `409`. This mode is locally verified only and is not enabled on public routing.
- Remote public proof must show Pay idempotency replay for intents and invoices, replay-safe webhook metadata, webhook lookup/replay behavior, and audit-hashed Pay events on public endpoints.

AI Gateway permission and audit:

- `ynx-ai-gatewayd` is the independent public AI service on port `6429`. `GET /health` exposes chain identity, provider configuration state, model name, rate-limit policy, build identity, and `truthfulStatus=chain-context-and-provider-backed-ai-gateway`; `GET /metrics` exposes request, success, denial, error, and active-request counters.
- Protected routes require `X-YNX-AI-Key` or `Authorization: Bearer <YNX_AI_GATEWAY_API_KEY>`. The provider key and access key are loaded only from the gateway's `0600` env file. In rollback-compatible `authoritative` mode, `YNX_AI_GATEWAY_UPSTREAM_KEY` authenticates gateway-to-chain proxy calls. In `bft` mode, exactly one process-local signer key source must match `YNX_AI_GATEWAY_SIGNER_ADDRESS`; the gateway serializes account nonce selection, signs canonical application actions locally, and sends only signed envelopes to the loopback BFT Gateway.
- `GET /ai/stream?session=...&q=...` calls the configured OpenAI-compatible provider with current YNX chain status context and streams isolated SSE `metadata`, `token`, and `done` events carrying a unique request ID. It does not execute value-moving or Trust-affecting actions.
- Gateway JSONL audit entries contain request/session/path/status/outcome, prompt hash, and audit hash. Raw prompts, access keys, provider keys, private keys, and seed phrases are not written.
- `POST /ai/permissions` creates an explicit scoped permission grant with `sessionId`, `requester`, `scope`, `purpose`, expiry, status, and `auditHash`; `GET /ai/permissions?sessionId=...` lists grants through the same authenticated service boundary.
- `POST /ai/actions` stores an AI action proposal with `sessionId`, `requester`, `scope`, `actionType`, `description`, sensitivity metadata, executable status, reasons, `auditHash`, and transparency entry.
- Sensitive AI actions include value movement, Trust label/risk-state changes, and sensitive-data or evidence export. They are persisted as `pending_approval` and `executable=false`.
- `POST /ai/actions/{id}/approve` requires a matching active permission for the same session, requester, and scope before a sensitive action can become `approved` and `executable=true`.
- `POST /ai/actions/{id}/reject`, `GET /ai/actions?sessionId=...`, and `GET /ai/actions/{id}` expose auditable review state without bypassing YNX Chain Law.
- BFT compatibility also exposes `GET /ai/permissions` and `GET /ai/audit`. BFT mutation responses carry signer, committed transaction hash, block height, block-time timestamps, and record audit hashes. The BFT Gateway verifies Comet hash, block membership, action type, signer, nonce, record ID, and final ABCI state before returning success.
- Remote public proof must show independent gateway health and release identity, authenticated SSE with request/session evidence, a sensitive action proposal remaining non-executable until a matching scoped permission approval, then active permission audit, approved executable state, and audited gateway lookup/list on public endpoints.
