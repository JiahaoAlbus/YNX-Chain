# Support, Security Reports, Disputes, and Refunds

Version: 1.0.0-candidate  
Status: workflow specification; public support URL and staffed service not evidenced

YNX operators will never ask for a seed phrase, private key, validator key, PEM file, CVV or complete API secret. Until an official support URL is published in `public-product-metadata.json`, users should not trust an unofficial contact claiming to recover funds or accounts.

Support intake should assign a case ID and collect only product, network, approximate UTC time, public transaction hash where applicable, non-secret request/error ID, user-visible message, and concise description. Sensitive evidence uses an approved private channel, not public issues or chain state.

Security reports require safe-harbor and disclosure terms approved by counsel, severity triage, acknowledgement target, encrypted evidence path, coordinated disclosure, remediation owner and reporter updates. Do not include exploit details on a public status page while containment is active.

Transaction disputes distinguish submission, provider acknowledgement, inclusion, settlement and application receipt. Final public-chain transactions may be irreversible. YNX cannot reverse a user-authorized canonical transaction merely because its outcome was unfavorable. Compromise, duplicate execution, provider mismatch and UI misrepresentation require evidence review.

Merchant refunds are new value-moving actions subject to merchant terms, payer/payee verification and authoritative settlement evidence. An application refund record is not proof that funds returned. Fees, timing, exchange-rate effects and unavailable routes must be disclosed before approval.

Appeals concerning Trust labels follow `docs/trust/TRUST_APPEALS_MARKET_INTEGRITY.md`. Privacy requests must distinguish deletable off-chain data from immutable public-chain records. Response-time promises remain unset until a staffed rota and routing test exist.
